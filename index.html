InnoDB存储引擎索引

发表于 2017-11-17 | | 阅读次数

概述

InnoDB中常见索引类型有如下三种：

  • B+树索引
  • 全文索引
  • 哈希索引
    哈希索引可以看成是对热点数据的映射，这个索引的管理是有InnoDB负责的，不需要DBA的干预。

    B+树索引

    B+树索引的本质就是B+树在数据库中的实现，由于B+索引在数据库中有一个特点是高扇出性（B+树一般只有2-4次），因此查询某一键值一般都要2-4次IO操作。

    聚集索引

    聚集索引就是按照每张表的主键构造一颗B+树，同时叶子节点中存放的即为整张表的行记录数据，也将聚集索引的叶子节点称为数据页。查询优化器倾向于采用聚集索引，因为聚集索引能在B+树索引的叶子节点上直接找到数据。
    注意：数据库中的记录是按照主键逻辑存储的，这其中有两个意思：1、页通过双向链表连接，页与页之间不是物理连续的；2、行记录通过双向链表连接，物理存储上可以同样不按照主键存储。

    辅助索引

    辅助索引的叶子节点并不包含行记录的全部数据，叶子节点出了包含键值外，还有一个书签（bookmark），该书签记录了哪里可以找到行数据。
    当通过辅助索引来寻找数据时，InnoDB会遍历辅助索引并通过页级别的指针获得指向主键索引的主键，然后通过主键索引找到一个完整的行记录。
    另外需要注意的是，B+树比较适用于高选择性的列。
    联合索引可以避免一次filesort排序操作。

    覆盖索引

    覆盖索引指的是从辅助索引中就可以得到查询的记录，而不需要查询聚集索引中的记录。使用覆盖索引的一个好处就是辅助索引不包含整行记录的所有信息，其大小远小于聚集索引，因此减少了大量的IO操作。

    MRR

    MRR工作方式如下：
    1、将查询到的辅助索引键值存放于一个缓存中，这时缓存中的数据根据辅助索引键值排序的。
    2、将缓存中的键值按照主键排序
    3、根据主键的排序顺序来访问实际的数据文件

    ICP优化

    未启用ICP（Index Condition Pushdown）时，数据库会先根据索引查询记录，然后根据where条件来过滤记录。启用后，数据库会在取出索引的同时，就判断是否可以进行where条件的过滤。
    举个栗子：
    某张表有联合索引(zip_code,last_name,first_name)

对于上述语句，MySQL数据库可以通过索引来定位zip_code=’95054’的记录，但是索引对于后两个条件没有帮助。，此时不启用ICP，则数据库会先通过索引取出所有zip_code=’95054’的记录，再过滤。
如果启用了ICP，则取出索引的时候就会对last_name这个条件进行过滤，不会对address条件过滤，因为address没有被包括在索引中。

undo log

我们通常说到事务时，都会谈起事务的ACID四大特性。undo log主要用来实现事务的原子性，举个栗子：

如果A-F之间出现宕机，那么G还没有执行，即磁盘中数据还没有被更新，此时是不会影响到事务的原子性的；如果在G-H出现宕机则说明undo log已经将事务开始时的数据状态记录到磁盘中了，但是数据的持久化出错，这样的话可以通过undo log将事务回滚到最初状态。

缺陷：每个事务提交前将数据和Undo Log写入磁盘，这样会导致大量的磁盘IO，因此性能很低。

redo log

和Undo Log相反，Redo Log记录的是新数据的备份，主要实现事务的持久性。在事务提交前，只要将Redo Log持久化即可，
不需要将数据持久化。当系统崩溃时，虽然数据没有持久化，但是Redo Log已经持久化。系统可以根据
Redo Log的内容，将所有数据恢复到最新的状态。

Undo + Redo的设计主要考虑的是提升IO性能。虽说通过缓存数据，减少了写数据的IO.
但是却引入了新的IO，即写Redo Log的IO。如果Redo Log的IO性能不好，就不能起到提高性能的目的。
为了保证Redo Log能够有比较好的IO性能，InnoDB 的 Redo Log的设计有以下几个特点：

A. 尽量保持Redo Log存储在一段连续的空间上。因此在系统第一次启动时就会将日志文件的空间完全分配。
以顺序追加的方式记录Redo Log,通过顺序IO来改善性能。
B. 批量写入日志。日志并不是直接写入文件，而是先写入redo log buffer.当需要将日志刷新到磁盘时
(如事务提交),将许多日志一起写入磁盘.
C. 并发的事务共享Redo Log的存储空间，它们的Redo Log按语句的执行顺序，依次交替的记录在一起，
以减少日志占用的空间。

InnoDB关键特性

发表于 2017-11-07 | | 阅读次数

InnoDB关键特性

Innodb关键特性主要有：

  1. 插入缓冲
  2. 两次写
  3. 自适应哈希索引
  4. 异步IO
  5. 刷新邻接页

    Insert Buffer

    首先，Insert Buffer和数据页一样是物理页的组成部分。InnoDB中主键是行唯一的标识符，通常应用程序中行记录的插入顺序是按照主键递增的顺序进行插入的。如果数据页中的行记录也是按照这个顺序进行存放的，那么这样的插入速度必然很快。这里提到的主键就是聚集索引，当然一张表中不可能只有一个聚集索引，还有其他的非聚集索引，一般称为辅助索引，如果用户按照非聚集索引进行查询操作，那么可能会在一段时间内访问多个数据页，这样会进行多次的IO操作。另外在插入数据时，我们除了需要生成主键索引树，还需要生成这些辅助索引树，由于插入的顺序是按聚集索引增长的顺序插入的。所以每插一个数据的时候，对于非聚集索引来说可能访问的都是不同的索引页，这样会增加IO消耗。
    为了解决这种问题，Innodb提出了Insert Buffer，对于非聚集索引的插入或更新操作，不是每一次直接插入到索引页中，而是先判断插入的非聚集索引页是否在缓冲池中，若在直接插入；若不在，则先放入到一个Insert Buffer对象中。然后按照一定的频率对Insert Buffer中的对象和辅助索引页节点进行合并操作，这时通常能将多个插入合并到一个IO操作中。
    使用Insert Buffer需要满足以下两个条件：
  6. 索引是辅助索引
  7. 索引不是唯一
    解释下为什么索引需要是非唯一的？因为对于唯一索引来说，每次插入需要查找索引页来判断插入记录的唯一性，这样的话每次插入都必须访问一次索引页，就没有该进的空间了。

    Insert Buffer的内部实现

    Insert Buffer的数据结构是一棵B+树，该B+树位于共享表空间中，因此如果仅使用独立表空间去恢复数据，那么可能还需要使用repair table去重新建立辅助索引。该B+树的非叶节点存放的是search key，结构如下：
名称 占用字节数 描述
space 4B 待插入记录所在表的表空间id
marker 1B 兼容老版本
offset 4B 表示页所在的偏移量

当一个辅助索引要插入到页时，如果该页不在缓冲池中，那么Innodb会构建一个上述结构的search key节点。接下来会查询该Insert Buffer这棵B+树，然后将这条记录插入到Insert Buffer B+树的叶子节点中。每个叶子节点结构如下：

名称 占用字节数 描述
space 4B 待插入记录所在表的表空间id
marker 1B 兼容老版本
offset 4B 表示页所在的偏移量
metadata 4B -
secondary index record 辅助索引内容

Innodb中使用Insert buffer bitmap页去追踪辅助索引页空间使用的情况。

Insert Buffer的合并

何时合并？

  • 辅助索引页被读取到缓冲池时
  • Insert buffer bitmap追踪到辅助索引页无可用空间
  • Master Thread

两次写

当数据宕机时，若Innodb正在将某个页写入到页表中，而此时只写了一部分，那么很有可能会出现部分写失效的情况。此时可以通过redo log回复数据，但是redo log记录的是对页的物理操作，比如物理地址为800的位置写入‘aaa’。因此当这个物理位置为800的所在页的磁盘空间坏了，此时就无法通过redo log恢复数据了。Innodb针对该问题提出了doublewrite方案，在应用redo log之前，将数据写入一个副本中，一旦出现上述的问题时，通过该副本还原该页，在进行数据恢复。
Innodb中doublewrite体系架构如图：

doublewrite由两部分组成，一部分是内存中的doublewrite buffer，大小为2M，另一部分是物理磁盘上共享表空间中连续的128个页，即2个区，大小同样为2M。在对缓冲池中脏页进行刷新时，并不是直接写入到磁盘中，而是先将脏页复制到doublewrite buffer中，之后通过buffer再分两次将数据写入到doublewrite的磁盘空间中，由于磁盘空间中的128个页是连续的，所以这个过程是顺序读写，开销不大。在完成doublewrite页的磁盘读写后，再将buffer中的页写入到各个表空间文件中。

自适应哈希索引

B+树查询效率和树的高度成正比，在生产环境中，B+树的高度为3-4层，故需要做3-4次的查询。
如果对某个页的连续访问（100次或记录数的1/16次），且访问该页的访问模式都一样（查询条件一样），那么Innodb会为该页建立哈希索引。建立哈希索引后，下次以同样的访问模式访问时，直接通过该哈希索引读取对应页。

异步IO

AIO定义：用户可以在发出一次IO请求后，连续发送多次IO请求，无需等待IO的完成。
Sync IO：用户每发送一次IO请求都必须等该IO结束才能发送下次IO请求。
AIO的另一个好处就是可以进行IO合并，比如查询（8，6）（8，7）（8，8）三条数据，同步IO需要三次IO操作，AIO则会将这三个请求合并成一个IO。

刷新邻接页

当刷新一个脏页时，Innodb会将该页的邻接页也刷新，这样对磁盘的操作就是顺序的了，也可以通过AIO将多个IO写入操作合并为一个IO操作。

数据库和数据库实例

数据库：物理操作系统文件和其他类型文件的集合。
数据库实例：MySQL数据库实例由后台线程和一个共享内存组成。共享内存可以被运行的后台程序共享。

InnoDB存储引擎

InnoDB存储引擎支持事务，将数据放在一个逻辑的表空间中，这个表空间就像黑盒一样由InnoDB存储引擎自身进行管理。InnoDB通过使用多版本并发控制（MVCC）提高并发性，并且实现了SQL标准的四种事务隔离级别，默认为REPEATABLE-READ级别，同时使用next-key locking的策略来避免幻读现象的发生。
对于表数据的存储，InnoDB存储引擎采用了聚集的方式，因此每张表的存储都是按照主键的顺序存放的。

InnoDB体系架构

先看下InnoDB体系架构图：

后台线程

由于InnoDB是一个多线程模型，所以后台有多个不同的后台线程处理不同的任务。

Master Thread

最核心的后台线程，主要负责将缓冲池中的数据异步刷新到磁盘，保证数据的一致性，包括脏页的刷新，合并插入缓冲(INSERT BUFFER)、UNDO页的回收等。

IO Thread

在InnoDB存储引擎中大量使用了AIO(Async IO)来处理写IO请求，这样可以极大提高数据库的性能。IO Thread的工作主要是负责这些IO请求的回调处理。其中重要的IO线程有insert buffer thread、log thread以及几个读写线程。

Purge Thread

该线程用于回收已经使用并分配的undo页。

Page Cleaner Thread

将之前版本中脏页的刷新操作都放入到单独的线程中来完成，这个线程就是Page Cleaner Thread。

缓冲池

InnoDB存储引擎是基于磁盘存储的，并将其中的记录按照页的方式进行管理，另外由于CPU速度与磁盘速度之间的鸿沟，基于磁盘的数据库系统通常使用缓冲池技术来提高数据库的整体性能。
数据库在读取页的时候，会首先去缓冲池读取，若缓冲池中没有该页，再去磁盘中读取。
对于数据库的修改操作，首先修改缓冲池中的数据页，然后以一定的频率刷新到磁盘上。

LRU List、Free List和FLush List

数据库中的缓冲池是通过LRU算法来管理的，不过这个LRU算法是经过改良了的。在InnoDB引擎中，LRU List中还加入了midpoint位置，新读取到的页虽然是最新访问的，但并不是直接放入到LRU列表的首部的，而是放入到LRU列表的midpoint位置。
为什么不使用朴素的LRU算法呢？
如果有一个SQL查询是大范围的页查询，那么经过这一次SQL查询之后，LRU List中的页可能会被全部刷出。这就把LRU中的热数据也刷出了，但是如果每次插入时，是插入到midpoint位置，只有多次读取相同页时才将该midpoint位置的数据页移到前端。这样的话，即使做SQL的大范围查询也不会将热数据刷出。
LRU List用来管理已经读取的页，但当数据库刚启动时，LRU列表是控的。这个时候页都放在FREE列表中。

重做日志缓冲

InnoDB内存中除了缓冲池外，还有重做日志缓冲（redo log buffer）。InnoDB引擎首先将重做日志信息先放入到这个缓冲区，然后按一定的频率刷新到重做日志文件中。

Checkpoint技术

由于缓冲技术的存在，所以必然会导致缓冲池中的页比磁盘中的页要新。这个时候需要按一定频率将新数据刷新到磁盘中完成持久化，若在刷新这个过程中出现宕机，那么未刷新到磁盘中的这部分数据就会丢失。所以InnoDB又采用了Write Ahead Log策略，即当事务提交时，先写重做日志，再修改页。当宕机时，可以通过重做日志来恢复。
checkpoint技术用于解决以下几个问题：

  • 缩短数据库恢复时间
  • 缓冲池不够时，将脏页刷新到磁盘
  • 重做日志缓冲不够用时，刷新脏页到重做日志文件中
    当数据库发生宕机时，数据库不需要将重做日志文件中的所有日志都恢复，需要恢复的应该是那些未被刷新到磁盘中的部分，这个时候我们需要标记这个位置，这样的话，checkpoint之前的数据不需要重新恢复，但是checkpoint之后的数据需要恢复。
    对于InnoDB引擎来说，其是通过LSN（Log Sequence Number）来标记版本的。而LSN是8字节的数字，单位字节。每个页中有LSN，重做日志中也有LSN，checkpoint也有LSN。Checkpoint所做的事情就是将缓冲池中的脏页刷到磁盘中，它决定了每次需要刷新多少页，每次从哪里取页，以及什么时间触发checkpoint。
    Sharp Checkpoint
    Sharp Checkpoint发生在数据库关闭时，将所有脏页刷新到磁盘中。
    Fuzzy Checkpoint
    每次刷新一部分脏页，而不是刷新所有的脏页。
  • Master Thread Checkpoint
    Master Thread Checkpoint中发生的checkpoint差不多是以每秒或每10秒的速度从缓冲池的脏页列表中刷新一定比例的页到磁盘中。
  • FLUSH_LRU_LIST Checkpoint
    因为InnoDB存储引擎需要保证LRU列表中需要有差不多100个空闲页可供使用。Page Cleaner Thread会检查LRU列表中是否有足够的空闲页，倘若没有足够空闲页，则需要将LRU列表尾端的页移除，如果被移除的页是脏页，则需要进行Checkpoint。
  • Async/Sync Flush Checkpoint指的是重做日志文件不可用的情况，这个时候需要强制将一些页刷新回磁盘中。若将已经写入到重做日志的LSN记作redo_lsn，若将已经刷新回磁盘最新页的LSN记作checkpoint_lsn，则可以定义：
    checkpoint_age = redo_lsn - checkpoint_lsn;
    async_mark = 0.75total_redo_log_file_size;
    sync_mark = 0.9    total_redo_log_file_size;
    当checkpoint_agesync_mark时，触发Sync Flush，从Flush列表中刷新足够的脏页回磁盘。
  • Dirty Page too much：脏页太多的情况，也会导致InnoDB存储引擎强制进行checkpoint。

    Master Thread工作方式

    Master Thread具有最高的线程优先级别，其内部由多个循环（loop）组成：主循环、后台循环、刷新循环、暂停循环。Master Thread会根据数据库的运行状态在上述四个循环中切换。
    主循环：主要包含两大操作：1、每秒操作 2、每10s操作。其中每秒操作包括日志缓冲刷新磁盘，即使这个事务还没有提交（总是）、合并插入缓冲（可能）、至多刷新100个InnoDB的缓冲池脏页到磁盘（可能）；如果当前没有用户活动，则切换到后台循环。每10s操作包括刷新100个脏页到磁盘中（可能）、合并至多5个插入缓冲（总是）、将日志缓冲刷新到磁盘（总是）、删除无用的Undo页（总是）、刷新100个脏页或者10个脏页到磁盘（总是）。

Spring容器技术内幕

AbstractApplicationContext是ApplicationContext抽象实现类，该类的refresh方法定义了Spring容器从配置文件的加载到Bean的实例化完成整个过程所执行的操作。

内部工作机制

下图描述了Spring容器从加载配置文件到创建出一个完整Bean的作业流程以及参与的角色。

首先，ResourceLoader从存储介质中装载配置文件，并将该配置文件所代表的信息用Resource表示；
然后，BeanDefinitionReader解析上述的配置信息，将配置文件中标签的节点元素解析成BeanDefinition（此时的BeanDefinition是半成品）并将其放到BeanDefinitionRegistry（类似于专门存放BeanDefinition的数据库，后续取Bean Definition都需要从这个地方获取）；
接下来，Spring容器会扫描BeanDefinitionRegistry中实现了BeanFactoryPostProcessor接口的Bean并调用其中的方法，主要完成以下工作：

  1. 对使用占位符的bean标签进行解析，获取最终值，这意味着对一些半成品的BeanDefinition加工形成成品
  2. 扫描BeanDefinition，利用反射找出实现了PropertyEditor接口的Bean，并将其注册到PropertyEditorRegistry。
    完成上述工作后，Spring容器从BeanDefinitionRegitry中获取加工后的BeanDefinition,使用InstantiationStrategy着手进行Bean的实例化操作。
    然后，Spring容器会使用BeanWrapper对Bean进行包装，主要完成属性的设置工作；
    最后Spring容器会扫描BeanDefinitionRegitry中的Bean后处理器，完成对Bean的最后加工，装配出一个准备就绪的Bean。

    BeanDefinition

    之前我们就提到BeanDefinition是配置文件中标签的节点元素在Spring容器中的内部表示。一开始BeanDefinitionReader将配置信息解析成半成品的BeanDefinition（占位符未解析成最终值），并将其放入BeanDefinitionRegistry中，这个BeanDefinitionRegistry就像是存储BeanDefinition的内存数据库。工厂后处理器会对半成品的BeanDefinition进行加工形成最终的BeanDefinition（占位符已解析成最终值）。

    InstantiationStrategy

    当形成最终的BeanDefinition后，我们需要使用InstantiationStrategy根据BeanDefinition创建一个Bean实例。InstantiationStrategy是一个策略接口，不同的实现类意味着提供了不同的创建实例的方法，比如使用CGLib动态的创建Bean的子类对象，或者使用SimpleInstantiationStrategy利用Bean实现类的构造函数创建Bean实例。
    使用InstantiationStrategy创建的Bean实例，并没有对属性进行设置，要想设置属性值需要使用BeanWrapper。

Spring Aop原理及实现

发表于 2017-10-20 | | 阅读次数

Aop

Aop的全称是面向切面编程，这是相对于OOP来说的，OOP中是面向对象编程。AOP和OOP解决的都是代码复用问题，只不过OOP通过继承来解决纵向的代码复用，而AOP通过动态代理技术解决横向的代码复用。具体的例子：Person和Teacher都有name属性，Person是Teacher的父类，所以Teacher继承了Person后就会直接具有name属性这就是纵向的代码复用；而横向代码复用就比较复杂，比如说日志管理、性能检测这个业务逻辑在不同的场景中可能是一样的不需要改动的，但是为了代码复用我们不可能将其在每一个场景中都写一遍，所以此时需要代理技术将这些业务逻辑抽出来并织入到目标场景中。

Aop的实现基础

JDK动态代理技术和CGLib动态代理技术。Spring中的AOP底层也是使用这两个技术实现的，只不过根据不同的配置选择了不同的实现。以上两个技术可见上篇文章。

AOP术语

  • Jointpoint（连接点）：程序执行的位置。Spring中这个概念用的少，基本上都是对PointCut操作。连接点由两个信息确定：用方法描述的程序执行点；表示该执行点方位的位置信息（比如Before、after等）
  • PointCut：业务中，需要织入横向逻辑的程序执行位置，注意：Pointcut不具备表示方位信息的能力。由于大部分Aop都是针对于方法织入的，所以可以将Pointcut描述为一个方法的集合，该集合中的每一个方法都是需要织入横切逻辑的。
  • Advice：横向逻辑的具体实现，并包含了位置信息。比如前置增强、后置增强等
  • Target：需要增强的目标类
  • Weaving：将Advice添加到Pointcut的过程
  • Proxy：织入后产生的代理类
  • Aspect：Pointcut+Advice

TreeMap介绍

TreeMap底层使用红黑树实现，而红黑树是一个接近平衡二叉查找树的树形结构。所以TreeMap是一个有序的集合，如果未指定比较器则默认使用自然排序对keys进行排序，否则使用指定的比较器对keys排序。
TreeMap中，基本操作的时间复杂度都在Log(n),不论是get、put还是remove。注意：对于任意一个有序的map，不论是否明确指定了比较器，该集合中的元素必须满足一致性相等的原则(即compare(a,b)==0和a.equals(b)返回相同的boolean值)。这个概念在Compareable和Comparator中提到了，虽然不满足该性质也不会报错，但会使有序集合的部分操作出现不符合预期的情况。
TreeMap是非线程安全的，所以要在多线程下使用集合，可以使用Collections#synchronizedSortedMap()包装下，或者使用并发包中提供的有序集合。

TreeMap的声明

和其他集合差不多，唯一的区别就是要实现NavigableMap。NavigableMap是一个SortedMap的子接口，主要提供了一些和有序集合相关的方法，比如floorEntry、lowerEntry等等。实现这个接口，表明TreeMap是一个有序的集合。

这是一个红黑树的节点，所以必须有left、right、parent的定义，另外还需要用一个boolean值表示这个节点的颜色即color。另外在该内部类中，很多的操作都使用到了树的遍历，比如getFirstEntry、getLastEntry和successor(t)等。

TreeMap中的主要操作

get(Object key)

该方法的具体实现是通过getEntry(key)；然后从中获取value值，所以我们需要看下getEntry(key)的方法。

先看没有指定比较器的get方法，树的遍历，元素比较方法是使用默认的自然排序。

public V put(K key, V value)

添加指定的key-value对到集合中，如果已存在key，则替代原有的value。

上述的添加操作有两种情况，第一种是比较器不为空时，另一种是比较器为空时使用自然排序。共同的操作都是先找到需要添加的元素的插入位置，如果已存在则替换。fixAfterInsertion就是对TreeMap底层的红黑树插入后做的调整工作，比如旋转和重新着色。这个旋转和重新着色的代码就不研究了，比较复杂，可以看看我之前写的红黑树，里面有介绍具体的旋转和着色步骤。
