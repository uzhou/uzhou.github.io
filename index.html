Netty服务端源码分析

发表于 2018-01-24 | | 阅读次数

服务端创建步骤

服务端具体代码可以参考上篇博客，这里就不列出来了。

  1. 创建ServerBootstrap实例。ServerBootstrap是服务端启动的一个辅助类，可以通过Build模式设置相关的链接参数。
  2. 设置并绑定Reactor线程池。一般是两个线程池，bossGroup一般用于TCP连接、登陆认证、selector轮询等；workGroup一般用于处理IO操作，以及一些简单的用户自定义的Handler处理。
  3. 创建并绑定NioServerSocketChannel。作为服务端，我们不需要显示创建NioServerSocketChannel，只用在ServerBootstrap创建的时候指定对应的服务端Channel类型就好。Netty内部会自动使用反射创建一个NioServerSocketChannel实例。
  4. 创建并初始化ChannelPipeline。这个过程也是Netty自动为我们执行的，当一个Channel被创建时，对应的Pipeline就会被自动创建。这里的Pipeline指的就是NioServerSocketChannel。Pipeline的实质就是一个Handler链，在这里主要包含headHandler->ServerBootstrapAcceptor->tailHandler。其中headHandler中有bind方法、connect方法等。当然这里可以通过ch.addLast()添加自定义的Handler或者一些编解码器。
  5. 绑定并监听端口，将NioServerScoketChannel注册到Selector上监听客户端连接。
  6. Selector轮询。Netty会分配一个boss线程专门用于Selector轮询，一旦有准备就绪的Channel就会将其交给对应Handler处理。服务端开始启动后，一般只有ServerSocketChannel一个channel注册在Selector上，所以有准备消息也应该是TCP连接的消息。ServerSocketChannel为该连接请求创建一个channel，绑定对应的EventLoop，并将其注册到Selector上，等待感兴趣的事情发生。
  7. 当轮循到感兴趣的事情后，如果是连接事件就和步骤6一样处理，如果是其他的事件如IO事件，则让该事件发生的Channel对应的EventLoop去处理对应的Handler事件。

源码分析

按照上述步骤，创建ServerBootstrap时，使用的是无参构造函数，这里主要是因为ServerBootstrap的参数太多，使用Build的构造方式更加的适合。下面我们看下不同的参数的不同作用。

Netty线程模型

首先需要介绍下Netty的线程模型，比较原始的线程模型就是一连接一线程的BIO线程模型，这个模型很蠢只能用于并发度十分低的情况。高级点的比如Reactor线程模型：

Reactor单线程模型


一个线程就可以支持所有的IO处理。但是瓶颈也是显而易见的！我们看一个客户端的情况，如果这个客户端多次进行请求，如果在Handler中的处理速度较慢，那么后续的客户端请求都会被积压，导致响应变慢！所以引入了Reactor多线程模型!

Reactor多线程模型


Acceptor专门用来监听接收客户端的请求;I/O读写操作由线程池进行负责;每个线程可以同时处理几个链路请求，但一个链路请求只能在一个线程中进行处理。

Reactor主从线程模型


Acceptor不再是一个单独的NIO线程，而是一个独立的NIO线程池；Acceptor处理完后，将事件注册到IO线程池的某个线程上；IO线程继续完成后续的IO操作；Acceptor仅仅完成登录、握手和安全认证等操作，IO操作和业务处理依然在后面的从线程中完成。

Netty中的Reactor线程池

主要是两个：bossGroup和workGroup。前者用于处理TCP连接建立以及一些登陆认证等；后者处理具体的IO事件。

NioServerSocketChannel

之后，需要指定一个Channel的类型让Bootstrap使用反射创建一个Channel，该Channel主要用于处理incoming connections。

TCP参数

主要包含IP地址和端口号，backlog大小。其中backlog大小表示内核为该socket排队的最大连接个数。

Handler

其中.handler表示设置ServerBootstrap父类的handler；.childHandler表示设置子类的handler。区别在于，childHandler是NioServerSocketChannel使用的，并且之后连接进来的channel也必须执行；而父类的handler则是每一个TCP连接进来时执行的。

绑定本地端口

执行bind函数时：

  1. 首先根据提供的ServerSocketChannel类型创建一个NioServerSocketChannel。它有两个参数，参数1是从父类的线程池中取出的一个NioEventLoop，该线程用于监听和接收客户端连接；参数2是workGroup，用于处理IO读写。
  2. NioServerSocketChannel的初始化。首先是设置Socket的参数和NioServerSocketChannel的附加属性；接下来是将AbstractBootstrap的Handler添加到NioServerSocketChannel中的Pipeline中；然后将服务端注册的Handler即ServerBootstrapAcceptor添加到pipeline中。
  3. NioServerSocketChannel的注册。首先判断当前线程是否是绑定的EventLoop，如果是则继续，否则将注册封装成task添加到EventLoop的执行队列中。接下就是注册操作，一开始设置interestOps为0不是16，这主要是因为注册操作的完成可以触发doBeginRead，该函数中重新修改了interesOps为16表示对客户连接的请求感兴趣。

客户端接入源码分析

当选择器Selector检测到新的准备就绪的Channel，会根据就绪位的不同执行不同的操作。假设监听到的客户端连接操作，会交由NioServerSocketChannel处理，channel内部聚合了Unsafe。不同的channel有不同的Unsafe实现，对于NioServerSocketChannel来说是NioMessageUnsafe，所以连接到来时，会调用NioMessageUnsafe的read方法。该方法主要是接收连接后，创建一个对应的NioSocketChannel。然后将启动时传入的childHandler加入到客户端SocketChannel的ChannelPipeline中，并设置TCP参数，最后注册SocketChannel到Selector中。当注册操作完成后，会由HeadHandler中的read方法将网络操作位修改为OP_READ.

IO

何为流，stream是指从源到目标的有序数据。java中以内存为中心，若内存到IO设备则是输出流，若从IO设备到内存则是输入流。

分类

输入和输出流

如上所述，从内存到IO则为输出，反之输入流。Java中，只能从输入流中取数据，不能放数据；只能向输出流中放数据不能从中取数据。

字节流和字符流

根据每次从流中取或者放数据的大小分：若每次操作一个字节的数据则是字节流，若每次操作一个字符即两个字节则是字符流。字符流一般用于文本数据的传输，而字节流则可以传输任何内容比如二进制数据。

节点流和处理流

可以直接从/向一个IO设备取/放数据的流称为字节流，一般根据不同的IO设备，节点流的处理是不一样的。处理流则是对节点流封装后的产物。这种设计模式是装饰模式，为了让上层应用在调用的时候更方便，消除不同节点流的差异。节点流的构造函数一般是以IO节点为参数，而处理流的构造函数则是以节点流为参数构造的。

处理流两个功能：

  • 性能提升：一般的节点流每次IO操作只能读取一个数据单元的数据，但处理流使用了缓冲区，并且OS层使用了DMA，cpu不用每次进行IO的系统调用都要“过问”一遍，极大提高了IO效率。
  • 操作方便：正是由于上述的变化，可以通过处理流一次读取一批的数据，而不是一个字节或者一个字符。

java1.4的NIO

上面的IO流，在读取不到数据时会让线程阻塞，不仅如此，底层的实现还是依赖于字节处理，面向流的输入输出每次只能处理一个字节的数据。

1.4NIO概述

NIO采用内存映射文件的方式来处理输入输出，NIO将文件或者文件中的一段映射到内存区域中，这样就可以像访问内存一样访问文件了。

channel

Channel和传统的InputStream、OutputStream最大的不同是Channel提供了一个map()方法，通过该方法可以将“一块数据”映射到内存中。另外Channel没有输入输出的概念，它是双向的，既可以取数据也可以放数据。

channel的使用

与传统流的两个区别：

  1. 直接将文件中内容映射成Buffer
  2. 程序不能直接访问channel中的数据包括数据的读取写入都不行，所有数据的操作都必须先经过Buffer

Buffer

Buffer的本质是一个数组，发送到Channel中的数据必须先放到这个Buffer中，从Channel中取数据时，Channel也必须先将数据放到Buffer中，然后从Buffer中取数据。

BUffer的使用

Buffer的四个概念：

  1. capacity：容量
  2. limit：位于limit后的区域既不能被读也不能被写
  3. position：用于指明下一个可以被读取或者写入的数据的索引位置。当使用Buffer从channel中读取数据，position表示已经读到了多少数据；当使用Buffer向channel中放数据时，则表示写了多少数据。
  4. mark：用于回滚的？

1.7NIO和1.4NIO

相比1.4的NIO来说，NIO2.0提供了基于异步channel的IO。可以说，1.4的NIO虽然是非阻塞的，但是仍然是同步非阻塞，到了1.7的NIO2.0才真正实现了异步非阻塞。

数据库中的锁用于管理对共享资源的并发访问，提供数据访问的完整性和一致性。

InnoDB的锁类型

首先我们需要知道InnoDB中的锁是行级锁，主要有两种类型：

  • S Lock：允许事务读一行数据
  • X Lock：允许事务删除或更新一行数据
    共享锁和排他锁的兼容性如下：

    另外InnoDB存储殷勤还支持意向锁，意向锁就是为了揭示下一行将要被请求的锁类型，意向锁为表级别的锁。
  • IS Lock：事务想获取一张表中某几行的共享锁
  • IX Lock：事务想获得一张表中某几行的排他锁
    一个事务要对某行加锁，则必须先对表加意向锁，如果该表已有锁且不兼容，则需要等待该表上的锁释放。

    一致性非锁定读

    一致性的非锁定读是指InnoDB存储引擎通过行多版本控制的方式获取当前执行时间数据库中行的数据。如果读取的行正在进行DELETE或UPDATE操作，这时读取操作不会因此去等待行上锁的释放。
    一致性的非锁定读原理图如下：

    由于undo段中保存了快照数据用于事务回滚，所以一致性的非锁定读中的快照数据没有额外的消耗。
    InnoDB会在事务隔离级别为READ COMMITED和REPEATETABLE READ下使用非锁定的一致性读提高并发性。只不过两者的处理方式不一样：READ COMMITED是读取最新的一份快照数据，而REPEATABLE READ读取的是事务开始时的行数据版本。

    一致性锁定读

    某些业务场景下，需要显示地对数据库读取操作进行加锁以保证数据逻辑的一致性。InnoDB存储引擎对于select语句支持两种一致性的锁定读操作：
  • SELECT … FOR UPDATE（对读取的行加X锁）
  • SELECT … LOCK IN SHARE MODE（对读取的行加S锁）
    上述两种锁都会在事务结束后被释放。

外键和锁

我们知道外键是用于保证参照完整性的，假如有表parent和child，其中child的第二列是外键，引用了parent。那么child每插入一次数据都要去parent中检查一下，看插入的数据是否存在，这个检查操作就是select。若此时parent表中被加了X锁，那么child的插入操作会被阻塞。这就是实际项目中为什么外键用的少的原因，是在影响并发性能，如果业务逻辑自身就能保证参照完整性那么就没有必要加外键。

锁带来的问题

脏读

脏页和脏数据的区别：
脏页是因为数据库内存中的数据页得到了修改，但是还没有刷新到磁盘上，这个时候的内存数据页被称为脏页。这并不影响数据的一致性，因为最终会通过异步刷新将脏页的修改刷新到磁盘上。而脏数据指的是某个事务未提交的数据。
脏读就是指，在不同的事务下，当前事务可以读到另外事务未提交的数据。脏读只有在事务隔离级别为READ UNCOMMITED情况下才发生，InnoDB默认隔离级别为REPEATABLE READ。

不可重复读

不可重复读是指在同一个事务内多次读取同一数据集合，而在这个事务还没有结束时，另一个事务也访问该同意数据集合，并做了修改，导致第一个事务多次读取的结果不一样。
不可重复读和脏读区别在于：脏读读到的是未提交的数据，而不可重复读读到的是另一个事务已经提交的数据。由此带来的问题被称为Phantom Problem即幻象问题。InnoDB通过next key lock算法，成功解决该问题。

丢失更新

一个事务的更新操作会被另一个事务的更新操作覆盖，从而导致数据的不一致。例如：
1）事务T1将行记录r更新为v1
2）事务T2将行记录r更新为v2
3）事务T1提交
4）事务T2提交
解决该问题可以将事务隔离级别设为可串行化。

阻塞

有时候，一个事务需要等待另一个事务释放锁资源后才能继续，否则就需要等待，这个等待就是我们说的阻塞。

死锁

死锁是指两个或者两个以上的事务在执行的过程中，因争夺锁资源而造成的一种互相等待的现象。InnoDB解决死锁的方法很简单就是超时回滚，即当出现死锁且超时，令其中某一个事务回滚。出了超时机制，当前数据库通常采用wait-for graph等待图）的方式来检测死锁的发生。

InnoDB存储引擎索引

发表于 2017-11-17 | | 阅读次数

概述

InnoDB中常见索引类型有如下三种：

  • B+树索引
  • 全文索引
  • 哈希索引
    哈希索引可以看成是对热点数据的映射，这个索引的管理是有InnoDB负责的，不需要DBA的干预。

    B+树索引

    B+树索引的本质就是B+树在数据库中的实现，由于B+索引在数据库中有一个特点是高扇出性（B+树一般只有2-4次），因此查询某一键值一般都要2-4次IO操作。

    聚集索引

    聚集索引就是按照每张表的主键构造一颗B+树，同时叶子节点中存放的即为整张表的行记录数据，也将聚集索引的叶子节点称为数据页。查询优化器倾向于采用聚集索引，因为聚集索引能在B+树索引的叶子节点上直接找到数据。
    注意：数据库中的记录是按照主键逻辑存储的，这其中有两个意思：1、页通过双向链表连接，页与页之间不是物理连续的；2、行记录通过双向链表连接，物理存储上可以同样不按照主键存储。

    辅助索引

    辅助索引的叶子节点并不包含行记录的全部数据，叶子节点出了包含键值外，还有一个书签（bookmark），该书签记录了哪里可以找到行数据。
    当通过辅助索引来寻找数据时，InnoDB会遍历辅助索引并通过页级别的指针获得指向主键索引的主键，然后通过主键索引找到一个完整的行记录。
    另外需要注意的是，B+树比较适用于高选择性的列。
    联合索引可以避免一次filesort排序操作。

    覆盖索引

    覆盖索引指的是从辅助索引中就可以得到查询的记录，而不需要查询聚集索引中的记录。使用覆盖索引的一个好处就是辅助索引不包含整行记录的所有信息，其大小远小于聚集索引，因此减少了大量的IO操作。

    MRR

    MRR工作方式如下：
    1、将查询到的辅助索引键值存放于一个缓存中，这时缓存中的数据根据辅助索引键值排序的。
    2、将缓存中的键值按照主键排序
    3、根据主键的排序顺序来访问实际的数据文件

    ICP优化

    未启用ICP（Index Condition Pushdown）时，数据库会先根据索引查询记录，然后根据where条件来过滤记录。启用后，数据库会在取出索引的同时，就判断是否可以进行where条件的过滤。
    举个栗子：
    某张表有联合索引(zip_code,last_name,first_name)

对于上述语句，MySQL数据库可以通过索引来定位zip_code=’95054’的记录，但是索引对于后两个条件没有帮助。，此时不启用ICP，则数据库会先通过索引取出所有zip_code=’95054’的记录，再过滤。
如果启用了ICP，则取出索引的时候就会对last_name这个条件进行过滤，不会对address条件过滤，因为address没有被包括在索引中。

undo log

我们通常说到事务时，都会谈起事务的ACID四大特性。undo log主要用来实现事务的原子性，举个栗子：

如果A-F之间出现宕机，那么G还没有执行，即磁盘中数据还没有被更新，此时是不会影响到事务的原子性的；如果在G-H出现宕机则说明undo log已经将事务开始时的数据状态记录到磁盘中了，但是数据的持久化出错，这样的话可以通过undo log将事务回滚到最初状态。

缺陷：每个事务提交前将数据和Undo Log写入磁盘，这样会导致大量的磁盘IO，因此性能很低。

redo log

和Undo Log相反，Redo Log记录的是新数据的备份，主要实现事务的持久性。在事务提交前，只要将Redo Log持久化即可，
不需要将数据持久化。当系统崩溃时，虽然数据没有持久化，但是Redo Log已经持久化。系统可以根据
Redo Log的内容，将所有数据恢复到最新的状态。

Undo + Redo的设计主要考虑的是提升IO性能。虽说通过缓存数据，减少了写数据的IO.
但是却引入了新的IO，即写Redo Log的IO。如果Redo Log的IO性能不好，就不能起到提高性能的目的。
为了保证Redo Log能够有比较好的IO性能，InnoDB 的 Redo Log的设计有以下几个特点：

A. 尽量保持Redo Log存储在一段连续的空间上。因此在系统第一次启动时就会将日志文件的空间完全分配。
以顺序追加的方式记录Redo Log,通过顺序IO来改善性能。
B. 批量写入日志。日志并不是直接写入文件，而是先写入redo log buffer.当需要将日志刷新到磁盘时
(如事务提交),将许多日志一起写入磁盘.
C. 并发的事务共享Redo Log的存储空间，它们的Redo Log按语句的执行顺序，依次交替的记录在一起，
以减少日志占用的空间。

InnoDB关键特性

发表于 2017-11-07 | | 阅读次数

InnoDB关键特性

Innodb关键特性主要有：

  1. 插入缓冲
  2. 两次写
  3. 自适应哈希索引
  4. 异步IO
  5. 刷新邻接页

    Insert Buffer

    首先，Insert Buffer和数据页一样是物理页的组成部分。InnoDB中主键是行唯一的标识符，通常应用程序中行记录的插入顺序是按照主键递增的顺序进行插入的。如果数据页中的行记录也是按照这个顺序进行存放的，那么这样的插入速度必然很快。这里提到的主键就是聚集索引，当然一张表中不可能只有一个聚集索引，还有其他的非聚集索引，一般称为辅助索引，如果用户按照非聚集索引进行查询操作，那么可能会在一段时间内访问多个数据页，这样会进行多次的IO操作。另外在插入数据时，我们除了需要生成主键索引树，还需要生成这些辅助索引树，由于插入的顺序是按聚集索引增长的顺序插入的。所以每插一个数据的时候，对于非聚集索引来说可能访问的都是不同的索引页，这样会增加IO消耗。
    为了解决这种问题，Innodb提出了Insert Buffer，对于非聚集索引的插入或更新操作，不是每一次直接插入到索引页中，而是先判断插入的非聚集索引页是否在缓冲池中，若在直接插入；若不在，则先放入到一个Insert Buffer对象中。然后按照一定的频率对Insert Buffer中的对象和辅助索引页节点进行合并操作，这时通常能将多个插入合并到一个IO操作中。
    使用Insert Buffer需要满足以下两个条件：
  6. 索引是辅助索引
  7. 索引不是唯一
    解释下为什么索引需要是非唯一的？因为对于唯一索引来说，每次插入需要查找索引页来判断插入记录的唯一性，这样的话每次插入都必须访问一次索引页，就没有该进的空间了。

    Insert Buffer的内部实现

    Insert Buffer的数据结构是一棵B+树，该B+树位于共享表空间中，因此如果仅使用独立表空间去恢复数据，那么可能还需要使用repair table去重新建立辅助索引。该B+树的非叶节点存放的是search key，结构如下：
名称 占用字节数 描述
space 4B 待插入记录所在表的表空间id
marker 1B 兼容老版本
offset 4B 表示页所在的偏移量

当一个辅助索引要插入到页时，如果该页不在缓冲池中，那么Innodb会构建一个上述结构的search key节点。接下来会查询该Insert Buffer这棵B+树，然后将这条记录插入到Insert Buffer B+树的叶子节点中。每个叶子节点结构如下：

名称 占用字节数 描述
space 4B 待插入记录所在表的表空间id
marker 1B 兼容老版本
offset 4B 表示页所在的偏移量
metadata 4B -
secondary index record 辅助索引内容

Innodb中使用Insert buffer bitmap页去追踪辅助索引页空间使用的情况。

Insert Buffer的合并

何时合并？

  • 辅助索引页被读取到缓冲池时
  • Insert buffer bitmap追踪到辅助索引页无可用空间
  • Master Thread

两次写

当数据宕机时，若Innodb正在将某个页写入到页表中，而此时只写了一部分，那么很有可能会出现部分写失效的情况。此时可以通过redo log回复数据，但是redo log记录的是对页的物理操作，比如物理地址为800的位置写入‘aaa’。因此当这个物理位置为800的所在页的磁盘空间坏了，此时就无法通过redo log恢复数据了。Innodb针对该问题提出了doublewrite方案，在应用redo log之前，将数据写入一个副本中，一旦出现上述的问题时，通过该副本还原该页，在进行数据恢复。
Innodb中doublewrite体系架构如图：

doublewrite由两部分组成，一部分是内存中的doublewrite buffer，大小为2M，另一部分是物理磁盘上共享表空间中连续的128个页，即2个区，大小同样为2M。在对缓冲池中脏页进行刷新时，并不是直接写入到磁盘中，而是先将脏页复制到doublewrite buffer中，之后通过buffer再分两次将数据写入到doublewrite的磁盘空间中，由于磁盘空间中的128个页是连续的，所以这个过程是顺序读写，开销不大。在完成doublewrite页的磁盘读写后，再将buffer中的页写入到各个表空间文件中。

自适应哈希索引

B+树查询效率和树的高度成正比，在生产环境中，B+树的高度为3-4层，故需要做3-4次的查询。
如果对某个页的连续访问（100次或记录数的1/16次），且访问该页的访问模式都一样（查询条件一样），那么Innodb会为该页建立哈希索引。建立哈希索引后，下次以同样的访问模式访问时，直接通过该哈希索引读取对应页。

异步IO

AIO定义：用户可以在发出一次IO请求后，连续发送多次IO请求，无需等待IO的完成。
Sync IO：用户每发送一次IO请求都必须等该IO结束才能发送下次IO请求。
AIO的另一个好处就是可以进行IO合并，比如查询（8，6）（8，7）（8，8）三条数据，同步IO需要三次IO操作，AIO则会将这三个请求合并成一个IO。

刷新邻接页

当刷新一个脏页时，Innodb会将该页的邻接页也刷新，这样对磁盘的操作就是顺序的了，也可以通过AIO将多个IO写入操作合并为一个IO操作。

InnoDB存储引擎

InnoDB存储引擎支持事务，将数据放在一个逻辑的表空间中，这个表空间就像黑盒一样由InnoDB存储引擎自身进行管理。InnoDB通过使用多版本并发控制（MVCC）提高并发性，并且实现了SQL标准的四种事务隔离级别，默认为REPEATABLE-READ级别，同时使用next-key locking的策略来避免幻读现象的发生。
对于表数据的存储，InnoDB存储引擎采用了聚集的方式，因此每张表的存储都是按照主键的顺序存放的。

InnoDB体系架构

先看下InnoDB体系架构图：

后台线程

由于InnoDB是一个多线程模型，所以后台有多个不同的后台线程处理不同的任务。

Master Thread

最核心的后台线程，主要负责将缓冲池中的数据异步刷新到磁盘，保证数据的一致性，包括脏页的刷新，合并插入缓冲(INSERT BUFFER)、UNDO页的回收等。

IO Thread

在InnoDB存储引擎中大量使用了AIO(Async IO)来处理写IO请求，这样可以极大提高数据库的性能。IO Thread的工作主要是负责这些IO请求的回调处理。其中重要的IO线程有insert buffer thread、log thread以及几个读写线程。

Purge Thread

该线程用于回收已经使用并分配的undo页。

Page Cleaner Thread

将之前版本中脏页的刷新操作都放入到单独的线程中来完成，这个线程就是Page Cleaner Thread。

内存

InnoDB内存结构图

缓冲池

InnoDB存储引擎是基于磁盘存储的，并将其中的记录按照页的方式进行管理，另外由于CPU速度与磁盘速度之间的鸿沟，基于磁盘的数据库系统通常使用缓冲池技术来提高数据库的整体性能。
数据库在读取页的时候，会首先去缓冲池读取，若缓冲池中没有该页，再去磁盘中读取。
对于数据库的修改操作，首先修改缓冲池中的数据页，然后以一定的频率刷新到磁盘上。

LRU List、Free List和FLush List

数据库中的缓冲池是通过LRU算法来管理的，不过这个LRU算法是经过改良了的。在InnoDB引擎中，LRU List中还加入了midpoint位置，新读取到的页虽然是最新访问的，但并不是直接放入到LRU列表的首部的，而是放入到LRU列表的midpoint位置。
为什么不使用朴素的LRU算法呢？
如果有一个SQL查询是大范围的页查询，那么经过这一次SQL查询之后，LRU List中的页可能会被全部刷出。这就把LRU中的热数据也刷出了，但是如果每次插入时，是插入到midpoint位置，只有多次读取相同页时才将该midpoint位置的数据页移到前端。这样的话，即使做SQL的大范围查询也不会将热数据刷出。
LRU List用来管理已经读取的页，但当数据库刚启动时，LRU列表是控的。这个时候页都放在FREE列表中。

重做日志缓冲

InnoDB内存中除了缓冲池外，还有重做日志缓冲（redo log buffer）。InnoDB引擎首先将重做日志信息先放入到这个缓冲区，然后按一定的频率刷新到重做日志文件中。

Checkpoint技术

由于缓冲技术的存在，所以必然会导致缓冲池中的页比磁盘中的页要新。这个时候需要按一定频率将新数据刷新到磁盘中完成持久化，若在刷新这个过程中出现宕机，那么未刷新到磁盘中的这部分数据就会丢失。所以InnoDB又采用了Write Ahead Log策略，即当事务提交时，先写重做日志，再修改页。当宕机时，可以通过重做日志来恢复。
checkpoint技术用于解决以下几个问题：

  • 缩短数据库恢复时间
  • 缓冲池不够时，将脏页刷新到磁盘
  • 重做日志缓冲不够用时，刷新脏页到重做日志文件中
    当数据库发生宕机时，数据库不需要将重做日志文件中的所有日志都恢复，需要恢复的应该是那些未被刷新到磁盘中的部分，这个时候我们需要标记这个位置，这样的话，checkpoint之前的数据不需要重新恢复，但是checkpoint之后的数据需要恢复。
    对于InnoDB引擎来说，其是通过LSN（Log Sequence Number）来标记版本的。而LSN是8字节的数字，单位字节。每个页中有LSN，重做日志中也有LSN，checkpoint也有LSN。Checkpoint所做的事情就是将缓冲池中的脏页刷到磁盘中，它决定了每次需要刷新多少页，每次从哪里取页，以及什么时间触发checkpoint。
    Sharp Checkpoint
    Sharp Checkpoint发生在数据库关闭时，将所有脏页刷新到磁盘中。
    Fuzzy Checkpoint
    每次刷新一部分脏页，而不是刷新所有的脏页。
  • Master Thread Checkpoint
    Master Thread Checkpoint中发生的checkpoint差不多是以每秒或每10秒的速度从缓冲池的脏页列表中刷新一定比例的页到磁盘中。
  • FLUSH_LRU_LIST Checkpoint
    因为InnoDB存储引擎需要保证LRU列表中需要有差不多100个空闲页可供使用。Page Cleaner Thread会检查LRU列表中是否有足够的空闲页，倘若没有足够空闲页，则需要将LRU列表尾端的页移除，如果被移除的页是脏页，则需要进行Checkpoint。
  • Async/Sync Flush Checkpoint指的是重做日志文件不可用的情况，这个时候需要强制将一些页刷新回磁盘中。若将已经写入到重做日志的LSN记作redo_lsn，若将已经刷新回磁盘最新页的LSN记作checkpoint_lsn，则可以定义：
    checkpoint_age = redo_lsn - checkpoint_lsn;
    async_mark = 0.75total_redo_log_file_size;
    sync_mark = 0.9    total_redo_log_file_size;
    当checkpoint_agesync_mark时，触发Sync Flush，从Flush列表中刷新足够的脏页回磁盘。
  • Dirty Page too much：脏页太多的情况，也会导致InnoDB存储引擎强制进行checkpoint。

    Master Thread工作方式

    Master Thread具有最高的线程优先级别，其内部由多个循环（loop）组成：主循环、后台循环、刷新循环、暂停循环。Master Thread会根据数据库的运行状态在上述四个循环中切换。
    主循环：主要包含两大操作：1、每秒操作 2、每10s操作。其中每秒操作包括日志缓冲刷新磁盘，即使这个事务还没有提交（总是）、合并插入缓冲（可能）、至多刷新100个InnoDB的缓冲池脏页到磁盘（可能）；如果当前没有用户活动，则切换到后台循环。每10s操作包括刷新100个脏页到磁盘中（可能）、合并至多5个插入缓冲（总是）、将日志缓冲刷新到磁盘（总是）、删除无用的Undo页（总是）、刷新100个脏页或者10个脏页到磁盘（总是）。
