Java设计模式--工厂模式

发表于 2017-05-31 | | 阅读次数

概述

工厂模式的目的就是为了将对象的创建从应用中隔离出来，从而降低耦合。主要体现了面向接口编程、依赖抽象而不是依赖具体类的原则。

简单工厂

工厂模式中，必然要涉及两个组件，一个是工厂组件，另一个则是需要创建的产品组件。我们按照对这两大组件的抽象程度可以将工厂模式分为简单工厂、工厂方法模式、抽象工厂模式。
简单工厂中，工厂组件是具体的，产品组件有抽象和实现的多个具体产品类。
这个工厂组件中一般有一个创建产品的具体方法（静态为主），该方法根据不同的需求返回不同的具体产品类对象。而凡是使用具体产品的业务逻辑只需要调用这个工厂对象的方法即可。
简单工厂模式将业务逻辑中可变的部分（根据不同的需求返回不同的产品对象）和不变的部分（对这些产品共有的操作）分离开来了。

工厂方法模式

工厂方法模式中，工厂组件有抽象的工厂和实现该抽象工厂的若干个工厂类，其中抽象工厂中包含一个创建产品对象的抽象方法，该方法必须在其子类得到实现，即产品的实例化被推迟到了子类；产品组件也有抽象产品和不同的实现类。

抽象工厂模式

抽象工厂与工厂方法不同在于工厂方法只能产生一个具体的产品对象，而抽象工厂的工厂组件是一个接口内有多个工厂方法，其实现类能够产生多个不同的产品对象。

概述

Annotation可以看成是一个对Java元素的标记，这个标记不会影响原Java程序的执行，但一般会被AnnotationProcessor解析并以此来执行其他操作。比如最常见的@Override注解，它的AnnotationProcessor就是编译器，编译器会查看该方法对应类的父类是否有对应的方法，如果没有就会报错，告知程序员父类中没有被重写的方法。

注解的使用，Annotation是一个接口，但是我们在定义自己的注解时，不能直接使用这个Annotation去声明。而只能使用@interface去声明，其实这个@interface已经隐式继承了Annotation了。

基本注解

@Override、@Deprecated、@SuppressWarnings以及Java 7出现的@SafeVarargs和Java 8中出现的@FunctionalInterface是5个基本的注解。
@Override修饰方法，表示该方法必须是重写父类的方法，否则编译出错。
@Deprecated用于表示某个程序元素(类或者是方法)已过时，当其他程序使用该元素时会给出警告。

@SafeVarargs用于抑制堆污染相关的警告。
@FunctionalInterface仅用于修饰接口，表示该接口是一个函数式接口，如果不是则会编译出错。

常见的元注解

元注解是用来修饰注解的注解，最常见的有@Retention、@Target和@Inherit三个。其中@Retention表示该注解的保留时间，若成员变量值为RetentionPolicy.RUNTIME,则表示这个注解能在运行时通过反射机制获取。@Target表示这个注解可以修饰哪些程序元素。@Inherit修饰的注解，表示被该注解修饰过的类的子类会自动被该注解修饰。

Java集合

Java集合基础

Collection

Collection是一个集合接口，与Map接口不同的是，实现Collection的集合类中的元素都是value型的，而实现Map接口的集合类的元素是型的。

Set

无序、不重复是Set集合的典型特征，但是Java中为了更好的满足现实需要，使得Set的部分实现类并没有完全遵循这个特征。

HashSet

利用hash算法实现内存存储，完全满足Set两大特征，无序不重复。其中如何判断两个元素是否重复呢？HashSet中判断元素是否相等，是通过两个函数equals和hashCode，必须equals返回true且hashCode返回值相同才能判定元素相等，元素相同时不能对HashSet进行重复插入。所以，当要将某个类的对象放入HashSet时，不要轻易重写上述两个函数，如果需要，则一定要保证其一致性。
HashSet不是线程安全的，所以当不同线程操作同一个集合时，一定要注意保证同步，Java中的Collections中提供了线程安全的方法。

LinkedHashSet——HashSet的子类

与HashSet不同的是，LinkedHashSet使用链表维护元素的次序，即它打破了无序性这个特征。

TreeSet

TreeSet保证了元素的排序状态，即它使元素在集合中按照某种升序或者降序排列。HashSet是采用hash算法来决定元素的存储位置，而TreeSet则是利用红黑树的数据结构来存储数据。由于TreeSet要对元素进行排序，所以放入TreeSet的对象必须实现Compareable接口。注意：每当添加一个元素到TreeSet中时，会将该元素和集合中的其他元素进行比较如果compareTo(Object obj)返回的是0，则表示这两个元素是相等的无法插入。另外，重写equals方法时，应该保证equals和compareTo方法具有一致性。

各Set总结

当不需要保持一个集合中元素的排序状态时，最好使用HashSet，因为HashSet性能更好，他不要维持红黑树的结构。LinkedHashSet在插入、删除不如HashSet，但是遍历效率却高得多。

List

一种长度可变的动态数组，与set不同的是List实现类允许有序和重复元素。

ArrayList

由于List可以被看成是一个长度可变的数组，所以Java引入了capacity和length的概念。当没有指定数组长度的时候，会给该数组指定一个初始的capacity，只有当数组的实际长度大于capacity时，才对数组进行重新分配更大的空间。ArrayList的实现结构是数组，所以在索引方面效率高但是插入删除较低。遍历时采用随机访问要比使用iterator迭代器要好。

Queue

Queue接口用于模拟队列这种数据结构，满足FIFO。

PriorityQueue

作为Queue的实现类，每次poll或者peek获取的元素是队列中最小的元素，严格来讲已经违背了FIFO的队列特征。由于要对元素进行排序，因此元素对应的类必须实现Compareable接口。

Deque

Queue的子接口，双端队列，还可以被当成栈来使用。

LinkedList

不仅实现了List接口还实现了Deque接口，所以LinkedList可以被当成栈使用。LinkedList内部是链表的实现方式，在插入删除效率上比ArrayList要高。遍历时采用iterator迭代器要比随机访问要好。

Map

Map的集合实现类存放的元素是键值对类型，前面介绍到的Set可以看成是value为null的特殊Map，因此在一些子类的实现上具有相同性质。

HashMap

可以将value看成是key的附属，这样，HashMap和HashSet就可以看成同一种结构。判断两个元素是否一致，也是看HashMap中key的equals和hashCode方法。

LinkedHashMap

使用双向链表维护插入顺序和迭代顺序一致。

TreeMap

红黑树结构，保证Map中的元素按照key的大小保持一种排序状态，所以元素对应的类需要实现Compareable接口。

深入理解JVM——虚拟机类加载机制

发表于 2017-04-09 | | 阅读次数

概述

Class文件对于JVM来说，只是JVM的原材料，真正对原材料进行加工的还得靠虚拟机的类加载机制。大概过程就是，JVM将Class文件加载到内存，并对数据进行校验、转换解析和初始化，最终形成虚拟机可以直接使用的java数据类型。

类加载过程

可以分为以下几个阶段：加载——>验证——>准备——>解析——>初始化——>使用——>卸载。前五个阶段是类加载阶段。

加载

加载所需要完成的功能：

  • 通过全限定名获取一个Class文件
  • 将这个Class文件中的静态存储结构转换为方法区的运行时数据结构
  • 在Java堆中生成一个代表该类的Class对象，作为方法区这些数据的访问入口

    验证

    验证是连接阶段的第一步，这一步是为了保证获取的Class文件能够满足JVM要求的数据格式，从而不会危害虚拟机自身的安全。
    1）文件格式验证
    2）元数据验证
    3）字节码验证
    4）符号引用验证

    准备

    这个阶段用于对类变量分配内存和赋予“零”值，至于这个变量的初始值则不是在这个阶段赋予的。

    解析

    这是一个将符号引用转换为直接引用的过程，但是它并不是将所有的符号引用都转成直接引用，一般是针对类和接口、字段、类方法、接口方法四类符号引用进行的。
    1、类或接口解析：通过全限定名和类加载器完成加载
    2、字段解析：先对该字段所属的类或者接口进行解析，然后在该类或接口中查找，若找不到，则现在该类或接口实现的接口中按照继承关系依次查找，否则在该类父类中查找
    3、类方法解析：首先解析当前类，然后在当前类中找，否则父类中递归查找，否则在实现的接口中递归查找
    4、接口方法解析：先解析当前接口，然后在当前接口中查找，否则在父类接口中找

    初始化

    类加载的最后一步。初始化阶段，JVM会按照程序的主观意愿通过一个()方法去初始化变量值，而这个方法是编译器自动收集类中的所有类变量的赋值动作和静态语句块中的语句合并而成的。

    类加载器

    类加载器是一段获取Class文件的代码模块，不同类加载器加载同一个类时，加载的类不是同一个类，也就是这两个类的对象使用instanceof关键字判断时，不是同一个类的对象。
    类加载中，最值得一提的是双亲委派模型：当要加载一个Class时，现将其扔给父类加载器加载，只有当父类加载器无法加载时，才自己去加载。
    从虚拟机的角度来说，只存在两种不同的类加载器：一种是启动类加载器（Bootstrap ClassLoader），该类加载器使用C++语言实现，属于虚拟机自身的一部分。另外一种就是所有其它的类加载器，这些类加载器是由Java语言实现，独立于JVM外部，并且全部继承自抽象类java.lang.ClassLoader。

从Java开发人员的角度来看，大部分Java程序一般会使用到以下三种系统提供的类加载器：
1)启动类加载器（Bootstrap ClassLoader）：负责加载JAVA_HOME\lib目录中并且能被虚拟机识别的类库到JVM内存中，如果名称不符合的类库即使放在lib目录中也不会被加载。该类加载器无法被Java程序直接引用。
2)扩展类加载器（Extension ClassLoader）：该加载器主要是负责加载JAVA_HOME\lib\，该加载器可以被开发者直接使用。
3)应用程序类加载器（Application ClassLoader）：该类加载器也称为系统类加载器，它负责加载用户类路径（Classpath）上所指定的类库，开发者可以直接使用该类加载器，如果应用程序中没有自定义过自己的类加载器，一般情况下这个就是程序中默认的类加载器。
我们的应用程序都是由这三类加载器互相配合进行加载的，我们也可以加入自己定义的类加载器。这些类加载器之间的关系如下图所示：

如上图所示的类加载器之间的这种层次关系，就称为类加载器的双亲委派模型（Parent Delegation Model）。该模型要求除了顶层的启动类加载器外，其余的类加载器都应当有自己的父类加载器。子类加载器和父类加载器不是以继承（Inheritance）的关系来实现，而是通过组合（Composition）关系来复用父加载器的代码。

Class类文件

Class类文件是通过javac编译器或其他方式编译成的一组以字节为基础单位的二进制流。该二进制文件描述了对应类的所有信息，包括类的访问权限控制，继承关系，实现接口信息，类成员，类方法以及方法中可能遇到的异常等等。与常规文件格式不同的是Class文件不是像XML等文件一样通过键值对记录数据的，Class文件对数据格式和顺序有着严格的要求，比如前四个字节是表示魔数的u4类型数据，再两个u2类型数据表示该Class文件对应版本号（不同jdk编译的Class文件，其版本号不一样，例如jdk1.6对应的是50.0）等等。

常量池之后的两个字节表示本类的访问控制权限，比如是PUBLIC类型的还是FINAL类型的，抑或是INTERFACE、ABSTRACT等等。

类索引、父类索引与接口索引集合

前两个数据项都是索引类型的，所以是一个u2类型数据，用于指向特定的目标内存地址，比如类索引中的u2数据指向该类的全限定名字面量的内存地址，父类索引则是指向父类的全限定名，接口索引是一个表数据，其中表数据中的前两个字节表示有多少个实现了的接口，接下来就是对应的接口索引。

深入理解JVM——JVM内存模型和垃圾收集

JVM内存模型

JVM基本概念介绍

  1. 什么是JVM？
    JVM是一个运行在操作系统之上的程序，通俗点说：这个程序向用户隐藏了屏蔽了与具体操作系统平台相关的差异性，使得经过javac编译器（或其他编译器）编译成功的Java字节码可以直接在JVM上运行。
  2. JDK、JRE、JVM是什么关系？
    首先，要知道JDK包含了JRE，而JRE包含了JVM。JDK是整个Java的核心，其中包括JRE、一堆Java工具和Java基础类库（Java API）等。而JRE（Java运行时环境）则包含了Java的核心类库和支持文件以及JVM。具体如下图所示：

内存模型

1、运行时数据区域
JVM在执行Java程序时，会将它管理的内存区域分成若干块，具体可以参照下图：

其中，我们只需要看Java堆、方法区、虚拟机栈（Java栈）、程序计数器、本地方法栈五个内存区域，其他的会在后续文章中介绍。
2、程序计数器
线程私有，和我们OS中的PC有一样的效果，不过这个指示的是当前字节码指令的行号。
3、虚拟机栈
线程私有，每一个线程对应一个虚拟机栈，这些虚拟机栈中元素是栈帧，而每一个栈帧对应一个方法，栈帧是方法运行时的内存模型，方法在内存中运行和栈帧在虚拟机栈中出栈入栈是保持一致的。每一个栈帧中一般会保存局部变量表、操作数栈、返回地址和动态链接等信息。
4、本地方法栈
线程私有，和虚拟机栈不同的是，它的元素是本地方法栈帧。
5、Java堆
所有线程共有，所有的实例对象都存储在堆内存中，另外GC管理的主要区域就是堆内存。
6、方法区
所有线程共有，存放类变量、常量、类信息；它还包含一个运行时常量池，主要存放一些字面量和符号引用、直接引用。

上述5个内存区域，除了PCR不会OOM，其他4个都可能会出现OOM异常。因此我们需要对内存中的垃圾进行回收。

JVM的垃圾收集

如何判断垃圾？

1、引用计数法
用一个数值表示一个对象的引用数量，一旦这个对象的引用计数为0则可以判定这个对象为死。无法解决循环引用的现象。
2、根搜索法
设置一系列的gc roots，当从这些gc roots向下搜索时，若找到对象，则表明这个对象是存活的，否则对象死亡。

4种引用

1、强引用
98%以上的都是这种，比如Person p = new Person()中的p为强引用。只要强引用存在，那么这个对象就不会被回收。
2、软引用
稍弱于强引用，当GC时，JVM不会第一时间选择回收弱引用，而是在回收完其他饮用厚，若还存在内存紧张的情况，那么会启动第二次GC回收软引用。
3、弱引用
只要遇到GC总是先回收弱引用。
4、虚引用
不用了解

三大类垃圾收集算法

标记类

1、标记-清除算法
直接对标记了的内存进行回收，主要缺点就是，产生大量的内存碎片，可能导致因无法申请到大内存而提前启动GC。
2、标记-整理算法
在1的基础上，进行内存压缩。

复制类算法

思想：将内存分为两块，块1为空，块2用于内存分配，当需要GC时，对块2GC并将存活下来的对象复制到块1中，调换块1、块2的逻辑。经典的是新生代的收集算法，将新生代内存分为了Eden、survivor1、survivor2，比例关系为8:1:1。主要缺点是浪费了10%内存。

分代收集算法

新生代复制算法、老年代标记类算法。

垃圾收集器

新生代收集器

1、Serial：单线程收集基于复制算法，特点是stop the world。Client模式下的JVM首选，即新生代较小的场景。
2、ParNew：和1唯一的区别是ParNew是多线程的。很受CPU个数的影响。
3、Parallel Scavenge：是一个“吞吐优先”的收集器，多线程基于复制算法。即：他没有缩短停顿时间，但是它把这个停顿时间分散开去了，使得在某一个时间段内，其吞吐量达到用户的需求。吞吐量=(用户线程运行时间)/(用户线程运行时间+停顿时间)。

老年代收集器

1、Serial Old：Serial的老年代版本，基于标记-整理算法。
2、Parallel Old：Parallel Scavenge的老年代版本。
3、CMS(Concurrent Mark Sweep)：基于标记-清除的并发收集器，该收集器有四个阶段，初始标记、并发标记、重新标记和清除。除了重新标记外其他阶段内都可以与用户线程并存，也就是说CMS缩短了停顿时间。
4、G1：将内存分为若干个大小一样的固定区域，根据每个区域的垃圾堆积程度，维护一个优先级列表，当需要进行GC时，就回收垃圾堆积最严重的那块内存。基于标记整理，另外它也缩短了停顿时间。

内存分配和回收策略

1、对象优先在Eden区分配
2、大对象直接进入老年代
3、长期存活的对象进入老年代
4、对象年龄可以动态判断，比如大于所有对象的平均年龄，则可以将其移入老年代
5、回收
1）年轻代和老年代
将内存划分的两个区域，一个叫年轻代、一个是老年代，其中年轻代中一般存放刚刚分配进来的对象，老年代中则是长时间存活或者十分巨大的对象。年轻代又被划分为Eden、survivor1、survivor2三个区域。
2）需要GC（Minor GC）时，会先在Eden中进行收集，并将不需要回收的对象移入survivor1中，survivor1将它的对象全部复制给survivor2，并调换survivor1和survivor2的逻辑顺序，若survivor2中的内存满了，则通过分配担保制度将1其移入老年代中。若老年代中的内存达到一个值时，我们就可以发启一次GC（Full GC，也就是Minor GC+Major GC）。
