先看一个图：

对于第一种插入方式：head作为一个ListNode的引用，他在进程栈中存放的是一个地址（假设是0x0010），执行temp=head时，temp会被赋值（0x0010），若此时执行的判断时temp!=null，那么最后会运行temp=null,此时temp中存放的不是任何一个地址，而是null。这就好比，temp此时和head单链表没有一点关系了，所以你后面的new ListNode()也就没有意义了。
对于第二种插入方式：前面都是一样的，不同的是判断了temp.next!=null，那么执行完while循环后，temp存放的是head单链表的尾节点的地址值，所以执行temp.next=new ListNode()之后，是能够将数据插进去的。

Remove Nth Node From End of List

发表于 2016-12-29 | | 阅读次数

脑子僵化了，怎么也想不出遍历一次的算法，总以为要先知道链表长度，才能准确定位到要skip的节点，后来才知道利用双指针可以很简单的解决这个问题。原理：删去倒数第n个节点，这个时候，被删的节点（我们假设是a）和链表的末尾（空节点）距离是知道的（距离为n），我们只要定义两个指针p1、p2，指针p2先移动n+1个距离。有人会问为什么是n+1个距离不是n，因为我们不是为了找到那个被删的节点，而是要删掉到，即跳过该节点。接下来就是同时移动指针p1、p2，直到p2指向链表末尾（指向null）。这个时候p1的位置就是要被删除的节点的前一个位置，最后只用p1.next=p1.next.next;这样就删除了这个节点。

Gradle生命周期

包括三个阶段：initialization、configuration和execution。
Initialization：这个时期，Gradle会决定哪些project参与到build中来，并为每一个project创建实例对象。
Configuration：在这个阶段配置每个Project的实例对象，比如将task表示成Gradle自己的model方便管理和后续操作。然后就执行这些项目脚本中的一部分task。
Execution：Gradle会执行那些在配置阶段已经创建和配置好的task子集，Gradle会将这些子集的名字作为参数传递给Gradle命令和当前目录。

Settings file设置文件

Gradle定义了一个设置文件，默认名称为settings.gradle在Gradle的初始化阶段(Initialization)加载settings.gradle文件。
多项目的工程settings.gradle文件在工程的根目录。
对于单个项目settings.gradle是可选择的，但是多个项目的工程它是必须要有的。
Settings.gradle定义了工程中包含的项目，你也有可能会添加一些库文件到脚本路径中。
对于一个构建脚本，属性和方法的调用被委托给一个Project类的实例对象。所以类似的settings.gradle中属性和方法被委托给一个Settings类的实例对象。
如果从settings.gradlea角度来看，initialization阶段做得就是找到settins.gradle并解析他，判断是多项目还是单项目。
