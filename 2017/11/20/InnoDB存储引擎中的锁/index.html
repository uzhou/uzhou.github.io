InnoDB存储引擎中的锁

发表于 2017-11-20 | | 阅读次数

什么是锁？
数据库中的锁用于管理对共享资源的并发访问，提供数据访问的完整性和一致性。

InnoDB的锁类型

首先我们需要知道InnoDB中的锁是行级锁，主要有两种类型：

  • S Lock：允许事务读一行数据
  • X Lock：允许事务删除或更新一行数据
    共享锁和排他锁的兼容性如下：

    另外InnoDB存储殷勤还支持意向锁，意向锁就是为了揭示下一行将要被请求的锁类型，意向锁为表级别的锁。
  • IS Lock：事务想获取一张表中某几行的共享锁
  • IX Lock：事务想获得一张表中某几行的排他锁
    一个事务要对某行加锁，则必须先对表加意向锁，如果该表已有锁且不兼容，则需要等待该表上的锁释放。

    一致性非锁定读

    一致性的非锁定读是指InnoDB存储引擎通过行多版本控制的方式获取当前执行时间数据库中行的数据。如果读取的行正在进行DELETE或UPDATE操作，这时读取操作不会因此去等待行上锁的释放。
    一致性的非锁定读原理图如下：

    由于undo段中保存了快照数据用于事务回滚，所以一致性的非锁定读中的快照数据没有额外的消耗。
    InnoDB会在事务隔离级别为READ COMMITED和REPEATETABLE READ下使用非锁定的一致性读提高并发性。只不过两者的处理方式不一样：READ COMMITED是读取最新的一份快照数据，而REPEATABLE READ读取的是事务开始时的行数据版本。

    一致性锁定读

    某些业务场景下，需要显示地对数据库读取操作进行加锁以保证数据逻辑的一致性。InnoDB存储引擎对于select语句支持两种一致性的锁定读操作：
  • SELECT … FOR UPDATE（对读取的行加X锁）
  • SELECT … LOCK IN SHARE MODE（对读取的行加S锁）
    上述两种锁都会在事务结束后被释放。

外键和锁

我们知道外键是用于保证参照完整性的，假如有表parent和child，其中child的第二列是外键，引用了parent。那么child每插入一次数据都要去parent中检查一下，看插入的数据是否存在，这个检查操作就是select。若此时parent表中被加了X锁，那么child的插入操作会被阻塞。这就是实际项目中为什么外键用的少的原因，是在影响并发性能，如果业务逻辑自身就能保证参照完整性那么就没有必要加外键。

锁的算法

  • Record Lock：单个行记录上的锁
  • Gap Lock：间隙锁，锁定一个范围，不包含记录本身
  • Next-Key Lock：Gap Lock+Record Lock（InnoDB对于行的查询采用的是这种算法）

锁带来的问题

脏读

脏页和脏数据的区别：
脏页是因为数据库内存中的数据页得到了修改，但是还没有刷新到磁盘上，这个时候的内存数据页被称为脏页。这并不影响数据的一致性，因为最终会通过异步刷新将脏页的修改刷新到磁盘上。而脏数据指的是某个事务未提交的数据。
脏读就是指，在不同的事务下，当前事务可以读到另外事务未提交的数据。脏读只有在事务隔离级别为READ UNCOMMITED情况下才发生，InnoDB默认隔离级别为REPEATABLE READ。

不可重复读

不可重复读是指在同一个事务内多次读取同一数据集合，而在这个事务还没有结束时，另一个事务也访问该同意数据集合，并做了修改，导致第一个事务多次读取的结果不一样。
不可重复读和脏读区别在于：脏读读到的是未提交的数据，而不可重复读读到的是另一个事务已经提交的数据。由此带来的问题被称为Phantom Problem即幻象问题。InnoDB通过next key lock算法，成功解决该问题。

丢失更新

一个事务的更新操作会被另一个事务的更新操作覆盖，从而导致数据的不一致。例如：
1）事务T1将行记录r更新为v1
2）事务T2将行记录r更新为v2
3）事务T1提交
4）事务T2提交
解决该问题可以将事务隔离级别设为可串行化。

阻塞

有时候，一个事务需要等待另一个事务释放锁资源后才能继续，否则就需要等待，这个等待就是我们说的阻塞。

死锁

死锁是指两个或者两个以上的事务在执行的过程中，因争夺锁资源而造成的一种互相等待的现象。InnoDB解决死锁的方法很简单就是超时回滚，即当出现死锁且超时，令其中某一个事务回滚。出了超时机制，当前数据库通常采用wait-for graph等待图）的方式来检测死锁的发生。
